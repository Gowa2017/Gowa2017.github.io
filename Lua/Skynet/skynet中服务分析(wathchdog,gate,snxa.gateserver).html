服务的三个阶段

  3. 工作阶段，收发消息了。

gate

watchdog 初始化后，即启动了 gate 服务；在收到 main 中的 start 指令后，它也会向 gate 发送一个 lua 消息，指令 open。

其本质，是把 snax.gateserver.lua加载到了 gate服务的虚拟机内，然后执行。

那么，背后的意义就是，wathchdog服务启动了一个gate服务，而且两者之间建立了联系。

gate 只是 gateserver 的一个使用，定义了对于 gateserver 处理的哪些消息要发到 gate 来进行处理。服务真正的启动，还是在 gateserver 模块内。gate 只是 gateserver 的消息处理器。

这样，消息在 snax.gateserver.lua代码中的 open处执行，开始监听端口。

消息处理流程

我们把gate 服务注册到了 skynet 中去后，在收到了对应的套接字消息后就会调用对应的回调函数进行处理。

一个典型服务器端流程是：

bind() -> listen() -> accept() -> read() -> response -> write() -> close()

客户端：

connect -> read()/write() -> close()

当我们注册了服务后，其实 gate 服务应该是处于 accept()状态，但这个工作由 skynet 完成。

gate 服务注册了两种消息类型的处理机制：

对于网络消息，定义在 MSG{}内，用来处理数据；而对于 lua 类型的消息，则定义在 CMD{}内，主要用来控制服务的启停。

完全不用关注 gateserver.lua中定义的那些函数，我们只关注一下我们自己定义的。

看来这个wathchdog只是一个接收各种事件的服务而已，并没有干什么卵事。

其还有一个重要的功能，就是每当一个新连接到来时开启一个 agent 服务。之后的消息 gate 就会全部都丢给 agent 处理了。

整个服务流程就是如下：

client connect() -> gate.connect() -> watchdog.open() -> new agent。
client data -> gate.data -> agent.dispatch()

架构图

事实上，Gate 只是作为 GateServer 的 Handler 而存在，两者共同提供服务。
然后 GateServer 收到 open 命令的时候，会将其指令来源，传递给 gate，我们这里是 WatchDog

digraph { WatchDog Gate GateServer Agent Client Client -> GateServer[label=Net] GateServer -> Gate[label=handler] Gate -> WatchDog }

因此，当有新客户端进行连接的时候

digraph { GateServer -> Gate[label="MSG.open",color=blue] Gate -> WatchDog[label="handler.connect",color=blue] WatchDog -> Agent[label="SOCKET.open",color=blue] Agent -> Gate[label="CMD.start",color=red] Gate -> GateServer[label="gateserver.openclient",color=red] }

之后，如果是有消息到达，都会由 Gate 重定向到 Agent 去。
